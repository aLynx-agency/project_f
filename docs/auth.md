# Auth

How authentication works in project_f — strategy, module layout, and the rules that prevent subtle bugs.

## Strategy

Google OAuth + magic link. No passwords.

Sign-in and sign-up are the same operation: Better Auth creates the user on first successful authentication. The UI never asks "are you new?" — it just asks for an email or a Google account. Routing after auth decides the experience:

| Condition                 | Destination         |
| ------------------------- | ------------------- |
| Existing user, any method | `/dashboard`        |
| New user, any method      | `/welcome`          |
| OAuth error (mid-flow)    | `/signin?error=...` |
| Magic link error          | `/error`            |

This is expressed via Better Auth's `callbackURL` / `newUserCallbackURL` / `errorCallbackURL` options on every `signIn.*` call — not via custom middleware logic.

## Module layout

```
src/lib/auth/
  auth.ts         Server-only. Creates the Better Auth instance. Imports Drizzle,
                  env vars, and plugins. Never import this from a client component.
  auth-client.ts  Client-safe. Creates the React auth client via createAuthClient().
                  Safe to import from any "use client" file.
```

There is no `index.ts` barrel in `src/lib/auth/`. That file was deleted — see below.

### Import rule

| Context                                | Import from              |
| -------------------------------------- | ------------------------ |
| Server components, route handlers, API | `@/lib/auth/auth.ts`     |
| Client components (`"use client"`)     | `@/lib/auth/auth-client` |

Never import from `@/lib/auth` (bare). Never re-export from a barrel that mixes both.

### Why no barrel

`auth.ts` imports `postgres` (the database driver). `postgres` uses Node's `fs` module internally. When a barrel re-exports both `auth.ts` and `auth-client.ts`, the bundler evaluates `auth.ts` as part of the client bundle — not because you imported anything from it, but because the module is referenced by the barrel. Tree-shaking cannot remove it because top-level imports have side effects.

The symptom is a hard build error:

```
Module not found: Can't resolve 'fs'
```

The permanent fix is no barrel. Every call site imports explicitly from the correct file.

## Plugins

| Plugin       | Purpose                                                                        |
| ------------ | ------------------------------------------------------------------------------ |
| `magicLink`  | Passwordless email sign-in. Calls `sendMagicLink()` from `src/lib/emails`      |
| `username`   | Stores a `username` + `displayUsername` alongside the email                    |
| Google OAuth | `socialProviders.google` — `accessType: "offline"`, `prompt: "select_account"` |

`emailAndPassword` is currently enabled in the Better Auth config but unused in the UI. It can be disabled once confirmed unnecessary.

## Email

Magic link emails are sent via Resend (`src/lib/emails/magicLink.tsx`).

- Template: `src/lib/emails/templates/MagicLink.tsx` — react-email JSX component with Tailwind styling.
- FROM: `Alynx <hello@alynx.agency>` — verified Resend domain with SPF/DKIM/DMARC.
- Dev fallback: when `RESEND_API_KEY` is unset, the link is logged to the console instead of sent. No email account needed for local development.

Preview the email template locally:

```bash
bun run email:dev   # preview server at http://localhost:3001
```

## Google OAuth Console setup

Each environment needs its own OAuth client:

1. Google Cloud Console → APIs & Services → Credentials → Create OAuth 2.0 Client ID
2. Application type: Web application
3. Authorized redirect URI: `${BETTER_AUTH_URL}/api/auth/callback/google`
   - Staging: `https://staging.hub.tryalynx.com/api/auth/callback/google`
   - Production: `https://hub.tryalynx.com/api/auth/callback/google`

## DB schema

Auth tables are generated by Better Auth and live in `src/db/schema/auth.ts`. They are re-exported from `src/db/schema/index.ts` alongside the app schema.

Migrations for auth tables:

- `0001_fast_masque.sql` — initial Better Auth tables
- `0002_nebulous_jazinda.sql` — `username` + `display_username` columns

Do not write raw `INSERT` statements for user seed data. Better Auth enforces invariants (password hashing, session linkage) that raw SQL bypasses. Create test users through the auth API instead.

## Route handler

`src/app/api/auth/[...all]/route.ts` — catches all Better Auth routes. No custom logic; Better Auth handles everything via the `auth` instance.
